Sonic Pad print farm — how it's wired & how it sounds

Three views of the build: the USB identity topology, the daemon's event→sound map, and the voice flow.

USB topology & printer identity

CH340 boards have no unique serial, so each printer is identified by the physical port it's plugged into (by-path), not the board. Three hang off the internal USB2 hub; Trident is on the separate CAM controller.

graph TD
  PAD["🖥️ Sonic Pad host"]
  PAD --> EHCI["ehci1 · USB2
internal 3-port hub"]
  PAD --> OHCI["ohci0 · USB1.1
CAM port"]
  EHCI -->|"port 1.1"| U["Unicorn · :7125"]
  EHCI -->|"port 1.2"| D["Dimeter · :7126"]
  EHCI -->|"port 1.3"| O["Omega · :7128 + :80"]
  OHCI -->|"port 0"| T["Trident · :7127"]
  classDef pad fill:#16181d,stroke:#0066ff,stroke-width:2px,color:#fff;
  classDef hub fill:#eef4ff,stroke:#0066ff,color:#16181d;
  classDef pr fill:#e9f9ef,stroke:#149e57,color:#16181d;
  class PAD pad; class EHCI,OHCI hub; class U,D,O,T pr;

Why it matters: a printer must stay on its labeled port — swap ports and you swap identities. That fourth path looks different (ohci0 vs ehci1) because it's a different controller — expected.

Event → sound map

The sonicpad-chimes.py daemon watches every printer's Moonraker event stream and plays a distinct sound per event — including emergency stop, which no macro can announce.

flowchart LR
  MR["Moonraker event streams
(all printers)"] --> D{"sonicpad-chimes.py
daemon"}
  D -->|"print started"| S1["▲ rising two-note"]
  D -->|"print complete"| S2["♪ four-note arpeggio"]
  D -->|"paused / runout"| S3["▼ falling two-note"]
  D -->|"resumed"| S4["▲ rising two-note"]
  D -->|"cancelled"| S3
  D -->|"error / M112 / shutdown"| S5["⚠ alternating alarm"]
  D -->|"(manual) boot"| S6["stock boot tone"]
  classDef src fill:#16181d,stroke:#0066ff,stroke-width:2px,color:#fff;
  classDef dae fill:#eef4ff,stroke:#0066ff,color:#16181d;
  classDef snd fill:#f6f8fb,stroke:#5a616e,color:#16181d;
  classDef alarm fill:#fff5e8,stroke:#e07d0a,color:#16181d;
  class MR src; class D dae; class S1,S2,S3,S4,S6 snd; class S5 alarm;

Themes: the same map drives any installed sound theme — switch soundscapes with ~/set-sound-theme.sh <name>; the daemon re-reads the active folder on every play. Families with variants (chime_done_a/b/c) rotate at random.

Voice flow — bank lookup vs. live synth

Every spoken line goes through say.sh. Fixed lines hit a pre-rendered WAV in the voice bank; only truly dynamic lines (times, percentages) are synthesized live. A single farm-wide lock plays one voice at a time — chimes bypass it and overlap freely.

flowchart TD
  A["say.sh [--force] <text>"] --> K["KEY = lowercase + underscores"]
  K --> Q{"~/voicebank/KEY.wav
exists?"}
  Q -->|"yes · fixed line"| LOCK
  Q -->|"no · dynamic line"| F{"first word =
a printer with a
Piper model?"}
  F -->|"yes"| PIPER["Piper live synth
(that printer's voice, nice'd)"]
  F -->|"no · Omega/System
or models deleted"| ESP["espeak-ng
(Omega robot voice)"]
  PIPER --> LOCK["take voice lock
flock /tmp/.voice_lock
→ one voice at a time"]
  ESP --> LOCK
  LOCK --> PLAY["play_chime.sh → aplay"]
  CH["🔔 chimes
(play_chime.sh direct)"] -. "bypass the lock,
overlap freely" .-> PLAY
  classDef entry fill:#16181d,stroke:#0066ff,stroke-width:2px,color:#fff;
  classDef dec fill:#eef4ff,stroke:#0066ff,color:#16181d;
  classDef bank fill:#e9f9ef,stroke:#149e57,color:#16181d;
  classDef synth fill:#fff5e8,stroke:#e07d0a,color:#16181d;
  classDef out fill:#f6f8fb,stroke:#5a616e,color:#16181d;
  class A entry; class Q,F dec; class LOCK,PLAY out; class PIPER,ESP synth; class CH bank;

Why it's built this way: pre-rendering fixed lines to WAVs means the bulky Piper models (~0.5 GB) can be deleted afterward — everything still speaks, since only dynamic lines need a synth and one lightweight espeak voice (Omega's) covers those. VOICES.md is the recipe to re-render any voice later.
